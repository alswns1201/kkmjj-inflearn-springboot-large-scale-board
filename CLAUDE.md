## Project Overview

This is a large-scale board system implemented as a microservices architecture using Spring Boot 3.3.2 / Java 21, designed to handle high traffic and large data volumes. It is an educational project focused on distributed system design patterns.

## Build and Test Commands

All Gradle commands must be run from the `kuke-board/` subdirectory (the Gradle root):

```bash
cd kuke-board

# Build all modules
./gradlew build

# Build a specific module
./gradlew :service:article:build

# Run all tests
./gradlew test

# Run tests for a specific module
./gradlew :service:article:test

# Run a single test class
./gradlew :service:article:test --tests "kuke.board.article.api.ArticleApiTest"

# Run a service locally
./gradlew :service:article:bootRun

# Clean build
./gradlew clean build
```

## Module Structure

```
kuke-board/
├── common/
│   └── snowflake/          # Distributed 64-bit ID generation (Snowflake algorithm)
└── service/
    ├── article/            # Article CRUD (port 9000, MySQL)
    ├── article-read/       # Read-optimized article queries
    ├── comment/            # Hierarchical comments (max 2 depth levels, port 9001)
    ├── hot-article/        # Daily popular article ranking (batch, runs at 1 AM UTC)
    ├── like/               # Like/upvote counting
    └── view/               # View count tracking (10-minute aggregation windows)
```

## Architecture

**Microservices with Database-per-Service:** Each service is independently deployable with its own datasource configuration.

**Distributed ID Generation (Snowflake):** All primary keys are generated by `common/snowflake` — a 64-bit ID composed of a timestamp epoch, node ID, and sequence number. The epoch starts at 2024-01-01 UTC. This is used instead of auto-increment to support horizontal sharding.

**Horizontal Sharding:** The `boardId` field on entities like `Article` acts as a shard key to distribute data across database instances.

**Layered Architecture per Service:**
- `controller/` — REST endpoints
- `service/` — business logic with `@Transactional`
- `repository/` — Spring Data JPA
- `entity/` — JPA domain models

## Local Development Prerequisites

- Java 21
- MySQL running on `localhost:3306`
- Create the `article` database before starting the article service: `CREATE DATABASE article;`

**Article service** connects to `jdbc:mysql://127.0.0.1:3306/article` (username: `root`, password: `root`).

**Schema management:** `ddl-auto: none` — schemas must be created manually (no auto-generation).

## Article Service API (port 9000)

```
POST   /v1/articles
GET    /v1/articles/{articleId}
PUT    /v1/articles/{articleId}
DELETE /v1/articles/{articleId}
GET    /v1/articles   (paginated list by boardId)
```

## Key Design Decisions

- **No Docker Compose** is configured; MySQL must be run independently.
- **`show-sql: true`** is enabled in `application.yml` — SQL queries are visible in logs during development.
- **`open-in-view: false`** — lazy loading outside transactions will throw exceptions; fetch data within service-layer transactions.
- The `ArticleApiTest` integration test calls a **live running service** via `RestClient`, not an embedded context — the service must be started before running it.
